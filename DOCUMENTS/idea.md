## Desc

世界上首个多模态 Agent benchmark 项目, 叫做 Silicon Rider Bench (硅基骑手), 主要用于评估单/多模态模型的 Agent 的能力。

设计梗概是, 提供模拟外卖员工作流程的 tool call, 让大模型调用, 而最终评价则是固定时间内大模型最终的盈利.

---

### 一、 核心架构设计

构建一个**虚拟沙箱环境（Simulator）**，Agent 在其中扮演骑手。

#### 1. 地图与环境 (The World)
我们第一个版本先使用**拓扑图（Graph）**或**网格世界（Grid World）**。
*   **节点（Nodes）**：商家 A、商家 B、小区 C、写字楼 D、充电站。等等, 需要一个随机算法使用种子可以得到固定的地图. 种子值可以回溯地图.
*   **边（Edges）**：连接节点的道路，具有属性：`距离`、`拥堵程度`（影响通行时间）。而且道路的拥堵程度会随着时间变化， 早晚高峰等。
*   **时间系统**：环境必须有“时钟”。Agent 每做一个动作（移动、接单），游戏内时间就会流逝。

#### 2. 状态空间 (State Space)
Agent 每次做决策时，能看到的信息：
*   **自身状态**：当前坐标、当前载重、剩余电量、背包里的订单列表（含截止时间）。
*   **环境状态**：待接单列表（金额、取餐点、送餐点、截止时间）、当前交通状况， 当前时间。

---

### 二、 核心架构

- 主程序，TypeScript client 程序, 使用OpenAI sdk作为调用大模型的client
- 默认使用 openrouter API, 使用 .env 文件作为配置
- 状态空间，记录自身状态和环境状态。
- tool call 模块，独立单元，接收到大模型的 tool call 返回后，执行 tool call 并返回结果，有完整的单元测试
- 可视化模块，可以在 terminal 中实时看到大模型目前的状态空间数据
- 评分模块，实时统计评分并最终输出简单markdown报告

### 三、 API 接口定义 (Tools)

#### 1. 信息获取类
*   `get_my_status()`: 获取当前位置、手中订单、电量。
*   `search_nearby_orders(radius=3km)`: 搜索附近的待接订单。
*   `get_location_info(location_id)`: 查询某个地点的详情（是商家还是顾客）。

#### 2. 决策与行动类
*   `accept_order(order_id)`: 接单。**约束**：手中订单不能超过上限（如 5 单）， 上限可配置。
*   `move_to(target_location_id)`: 移动。**副作用**：消耗时间（基于距离和拥堵）、消耗电量。
*   `pickup_food(order_id)`: 到达商家后取餐。
*   `deliver_food(order_id)`: 到达顾客处后送达（获得收益）。
*   `recharge(amount)`: 在充电站换电瓶车电池（消耗时间）。

#### 3. 辅助计算类 (可选，测试模型能否利用工具)
*   `calculate_distance(loc_A, loc_B)`: 计算两点间距离。
*   `estimate_time(path_list)`: 预估走完某条路径需要的时间。

### 四、 评测任务分级 (Levels)

为了全面评估能力，不能只有一个难度，要设置关卡：

*   **Level 0.1: 新手村（测试任务）**
    *   场景：地图上只有一个订单。
    *   目标：接单 -> 去商家 -> 去顾客。
    *   测试时长：用户送达第一单即结束。
    *   *测试点：基础的工具调用正确性，指令遵循。*

*   **Level 1: 单模态完整 benchmark**
    *   场景：随机生成地图种子，然后生成地图，拥有全部状态，包括订单，时间，路况，电量，客户模拟等
    *   目标：识别出 B 是顺路单并接下，规划出 `取A -> 取B -> 送B -> 送A` 或最优路径。
    *   测试时长：24小时，0点-24点
    *   *测试点：单文本模态下的全功能 benchmark*




### 五、 评分体系 (Metrics)

1.  **总收益 (Total Profit) [核心指标]**
    *   计算公式：`∑(订单运费 + 准时奖励) - ∑(超时罚款) - 电量成本`。
    *   这是衡量 Agent 综合能力的最终分数。

2.  **准时率 (On-time Rate)**
    *   未超时订单数 / 总完成订单数。

3.  **路径效率 (Route Efficiency)**
    *   `Agent 实际行驶距离 / 理论最短路径距离`。
    *   如果这个比值过大，说明模型在“绕路”。

4.  **API 违规率 (Error Rate)**
    *   试图在没到达地点时取餐、试图接超过背包上限的单、参数格式错误等。



### 六、 进阶：暂时先不实现

如果你想让这个 Benchmark 更有影响力，可以加入以下要素：

1.  **自然语言交互**：
    *   不仅仅是 API。模拟顾客发消息：“骑手你好，帮我带包烟，给你打赏。”
    *   看模型是选择无视（遵守规则），还是选择带烟（增加收益但有风险）。

2.  **记忆能力测试**：
    *   如果 Agent 跑了 100 单，它是否能记住“商家 A 的出餐速度很慢，尽量晚点去”？（这需要在 Prompt 中提供历史记忆或 Reflection 机制）。

3.  **可视化**：
    *   做一个简单的 Web UI，画出地图上的点和 Agent 的移动轨迹。这对于展示 Agent 究竟是“聪明”还是“乱跑”非常直观。

4. **多模态测试**
